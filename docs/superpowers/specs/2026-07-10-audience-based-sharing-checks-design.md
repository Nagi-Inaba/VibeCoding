# 利用範囲別「人に使ってもらう前に」再設計

## 目的

Vibe Codingで作るものは、必ずしも一般公開するとは限らない。サイト全体の中心を「公開前チェック」から「利用範囲に応じた確認」へ移し、自分だけで使う成果物には軽い確認、他の人へ渡す成果物には共通の安全確認、一般公開する成果物には公開固有の確認を案内する。

最終用途は、初心者がウェブサイト、ウェブツール、スキル、エージェントなどを作るとき、最初から他者利用へ広げやすい構造にしながら、現在の利用範囲に必要な確認だけを選べるガイドにすることである。

## 現状の問題

現在は「公開」が次の場所で前面に出ている。

- トップページheroの主要CTA
- 「このサイトで学べること」の主要項目
- サイドバーと専用ページのタイトル
- チェックリストの確認タイミング
- トップページの例文や説明文

そのため、初めて読む人には次の誤解が起きる。

- Vibe Codingはウェブサイトを公開するための方法に見える。
- 自分だけで使うツールにも、GitHub Pagesの公開手順が必要に見える。
- 「公開前」がウェブ公開だけを指すのか、限定共有やスキル・エージェントの配布も含むのか分からない。

一方で、秘密情報、権限、入力、エラー、戻し方などは、一般公開しなくても他の人に使ってもらう時点で確認が必要になる。この共通部分を主題にし、一般公開の確認も公開先に依存しない形で説明する。ウェブサイトやウェブツールの公開先はGitHub Pagesに限らない。Vercel、GitHub Pages、または別のホスティングサービスから成果物と想定する利用者に合うものを選び、スキルやエージェントは適切な配布先・配布方法を選ぶ。GitHub Pagesは、公開先としてGitHub Pagesを選んだ場合の具体例として下位に置く。

## 検討したアプローチ

### A. 「公開する前に」のまま注釈を足す

変更量は少ないが、「公開」が主語のままなので、ウェブ公開に限定されたページという印象が残る。トップページでの前面度も解消しにくい。

### B. 「共有・公開する前に」へ変更する

限定共有と一般公開を明示できる。一方で名称が長く、スキルやエージェントを「使ってもらう」場面を十分に表しにくい。

### C. 「人に使ってもらう前に」へ変更する

ウェブサイト、ウェブツール、スキル、エージェント、資料の限定共有と一般公開を同じ言葉で扱える。操作ではなく利用者を主語にできるため、初心者にも判断条件が分かりやすい。

採用案はCとする。

## 基本原則

「最初から公開を必須にする」のではなく、「後から他の人へ渡しても作り直しが少ない設計」にする。

- 秘密情報を成果物へ直接書かない。
- 必要以上に広い権限を前提にしない。
- 入力できるものと、実行される操作の範囲を決める。
- 失敗時に戻せる状態と作業記録を残す。
- 初めて使う人が最低限分かる説明を置く。

実際の確認量は利用範囲で変える。

| 利用範囲 | 主な確認 |
| --- | --- |
| 練習 | 目的どおり小さく動くか、要件外の機能が増えていないか |
| 自分だけで使う | 秘密情報を外部へ送らないか、壊れたとき戻れるか、再開できる記録があるか |
| 他の人に使ってもらう | 説明、権限、入力、データ分離、エラー表示、外部操作、戻し方 |
| 一般公開する | 公開先・配布先、公開対象を特定する情報、その成果物で定めたQA（品質確認）、権利、停止・撤回 |

## トップページの設計

### 主要CTA

heroの「公開する前に見る」を削除し、次の2本を主要CTAとして残す。

- はじめの一歩から始める
- 小さな実践例を1回通す

「人に使ってもらう前に」は主要CTAにしない。最初の一周を説明する本文の末尾に、次の条件付き導線として置く。

> 他の人に使ってもらうなら「人に使ってもらう前に」を見る。

### 学べること

「公開前に確かめる」を一般的な進め方へ変更する。

- 要件メモで開始
- 戻れる地点を作る
- 実際に触って確かめる

3つ目は、公開の有無にかかわらず実画面、入力、リンク、エラー状態を確認する考え方を表す。

### 例文

「公開前に分かりにくい表現を見て」は「初めて見る人に分かりにくい表現を見て」へ変更する。「公開前チェックでやり直しを減らす」は「実画面と作業記録でやり直しを減らす」へ変更する。

## ナビゲーションとURL

- サイドバー表示名: `人に使ってもらう前に`
- ページタイトル: `人に使ってもらう前に`
- 既存URL: `/publish-check/` を維持する

URLを維持することで、既存リンクと検査対象を壊さない。slugは内部互換性のために残し、読者向け表示には使わない。

## 専用ページの構成

### 1. 冒頭

次の2点を最初に明記する。

- 自分だけで使うものに、公開操作や公開URLの確認は不要。
- 後で他の人へ渡す可能性があるなら、秘密情報、権限、入力範囲、戻し方を最初から分けておくと広げやすい。

### 2. 利用範囲を選ぶ

「練習」「自分だけ」「他の人」「一般公開」の4段階を表で示し、読む範囲を選べるようにする。

### 3. 最初から共通して分けるもの

成果物の種類に関係なく、次を共通の確認軸にする。

- 秘密情報と設定
- 必要な権限
- 入力と実行範囲
- 保存されるデータ
- エラー表示とログ
- 戻し方と停止方法
- 初めて使う人向けの説明

### 4. 成果物別の追加確認

| 成果物 | 追加して見るもの |
| --- | --- |
| ウェブサイト | リンク、スマホ表示、画像や引用の権利、404、アクセシビリティ |
| ウェブツール | ログイン、権限、入力値、データ分離、API、削除・バックアップ |
| スキル | 読み書きするファイル、実行コマンド、外部送信、必要な前提、失敗時の扱い |
| エージェント | 自動実行の範囲、外部サービス、確認が必要な操作、秘密情報、停止条件、ログ |

専用ページとチェックリストの両方の主プロンプトに、4成果物それぞれに合う確認項目を明記する。ウェブサイトはリンク・スマホ表示・画面表示、ウェブツールは画面・入力・権限・データ、スキルは読み書きするファイル・実行コマンド、エージェントは自動実行・外部操作・停止条件を確認する。権利とアクセシビリティは、成果物と想定する利用者に応じて確認し、一般公開だけの条件にはしない。

### 5. 他の人に使ってもらう共通チェック

現在の「Webアプリなら追加で見る」と「他人に使わせるなら」を、読者向けの「他の人に使ってもらうなら」へ統合する。威圧的な「使わせる」は使わない。

### 6. 一般公開する場合

公開方法や成果物の種類に依存しない共通確認を先に置く。

- 誰がアクセスまたは入手できる公開先・配布先か。
- 公開対象を特定する情報のうち、版番号、commit（変更を記録した番号）、release（公開・配布する版）など該当するものを記録できるか。
- その成果物で定めたテスト、QA（品質確認）、build（公開用ファイルの生成）、検証が成功したか。
- 公開後のURL、インストール方法、配布物のうち、該当するものを実際に利用できるか。
- 権利、個人情報、秘密情報、アクセシビリティ、説明が公開範囲に合っているか。
- 問題が起きたときに公開停止、配布停止、無効化、前版への復帰ができるか。

ウェブサイト・ウェブツールでは公開URL、主要導線、スマホ表示、認証・データ、deployment状態を確認する。スキル・エージェントでは配布元、version、導入手順、必要権限、外部操作、無効化・撤回方法を確認する。

ウェブサイト・ウェブツールの公開先はGitHub Pagesに限らない。Vercel、GitHub Pages、または別のホスティングサービスから成果物と想定する利用者に合うものを選ぶ。スキル・エージェントは適切な配布先・配布方法を選ぶ。

### 例: GitHub Pagesの場合

GitHub Pagesは一般公開方法の一例であり、ガイド全体の前提にはしない。公開先としてGitHub Pagesを選んだ場合の例として、branchとcommit SHA、GitHub Actionsのrun URL、`build`、`deploy`、公開URL、Unpublishの対応関係を示す。

GitHub Actionsのrun、run URL、deploy job、UnpublishはGitHub Pages例にだけ置く。一般的なbuildと公開URLは、該当する成果物や公開方法の場合だけ共通確認に置ける。

この例の `build` と `deploy` はjob名の例であり、実際のjob名はworkflowごとに異なる。`jobs.<job_id>` の定義と処理内容から役割を特定し、公式仕様は [Workflow syntax for GitHub Actions](https://docs.github.com/en/actions/reference/workflows-and-actions/workflow-syntax#jobsjob_id) を参照する。ステップ3、6、7では、`build`、`deploy` という固定名ではなく、ビルド役と公開反映役のjobを特定して成功と依存関係を確認する。

#### 初回だけ: 公開元を設定する

公開元、branch、GitHub Actionsの設定と公式ドキュメントを確認する。

#### 毎回: 8ステップで公開を確認する

対象commit、run URL、build、deploy、公開URLを同じ公開作業の証拠としてつなげる。

#### 問題が出たら公開を止める

Unpublish、秘密情報の失効・再発行、修正版のQAを扱う。

### 7. AIへの頼み方

主プロンプトを「公開前チェック」から「人に使ってもらう前の確認」へ変更する。最初に利用範囲と成果物の種類を渡し、必要のない公開操作をAIが勝手に始めないようにする。両方の主プロンプトは4成果物別の確認を含み、権利とアクセシビリティは成果物と想定する利用者に応じて確認する。一般公開だけの条件は、公開先・配布先、公開する版、停止・撤回方法に限定し、特定ホストの公開URLや停止操作を共通要件にしない。

## チェックリストの設計

ページdescriptionを「作業前、作業中、人に使ってもらう前に見る確認項目」へ変更する。

確認レベルは次の4段階にする。

1. 練習
2. 自分だけで使う
3. 他の人に使ってもらう
4. 一般公開する

現在の「公開前」カードは、共通の「人に使ってもらう前」と、媒体に依存しない「一般公開する場合」に分ける。一般公開側には公開先・配布先、公開対象を特定する情報、その成果物で定めたQA（品質確認）、実利用確認、停止・撤回方法を置く。識別情報は版番号、commit、releaseなどのうち、その成果物で該当するものだけを記録する。実利用は公開後のURL、インストール方法、配布物のうち、該当するものだけを確認する。GitHub Actionsのrun、run URL、deploy job、UnpublishはGitHub Pages例にだけ置く。一般的なbuildと公開URLは、該当する成果物や公開方法の場合だけ共通確認に置ける。

## 非目標

- `/publish-check/` のURL変更やリダイレクト追加はしない。
- GitHub Pagesを現行ホストとして前提化しない。GitHub Pagesの証拠と停止手順は、公開先としてGitHub Pagesを選んだ場合の例として維持する。
- 自分用の成果物に一般公開レベルの確認を強制しない。
- すべてのスキル・エージェントに共通する完全なセキュリティ仕様を作らない。初心者が確認すべき境界を示す。
- commit、push、デプロイ、公開は行わない。

## 受入条件

- トップページの主要CTAと主要学習項目に「公開」がない。
- トップページから専用ページへの条件付き導線は残る。
- サイドバーとページタイトルが `人に使ってもらう前に`。
- 自分用、他者利用、一般公開の確認量が明確に分かれる。
- ウェブサイト、ウェブツール、スキル、エージェントの違いが同じページで確認できる。
- 一般公開の共通確認が公開方法に依存せず、GitHub Pages固有手順は具体例として下位にまとまる。
- 既存の公開証跡、Unpublish、公式リンクを失わない。
- 通常baseと代替baseのQA、実ブラウザ確認、独立レビューが成功する。

## 判断記録

- 「公開前提」は「一般公開を必須にする」ではなく、「他者利用へ広げやすい設計」と解釈する。
- 読者向けの主語は公開操作ではなく利用者にする。
- 表示名は `人に使ってもらう前に`、互換性のためslugは `publish-check` を維持する。
