## まず楽をする

バイブコーディングで大事なのは、いきなり自分だけで作り切ろうとしないことです。

- 誰かが作っているものを真似る
- 既存の公開スキルやエージェントを使う
- オープンソースのツールを使う
- 企業が公式に出しているドキュメントをAIに読ませる
- 繰り返す作業はスクリプトにして保管する

探す場所として一番使いやすいのは、やはりGitHubです。Google検索で自力で探し回るより、まず [GitHub CLI](https://cli.github.com/) を入れて、CodexやClaudeから `gh` コマンドを使える状態にしておく方が楽です。

GitHub CLIを使えるようにしておくと、たとえば次のことができます。

1. `gh repo clone` でGitHub上のリポジトリを簡単に手元へ持ってくる
2. `gh search repos` で、条件に合うリポジトリをAIに探させる
3. GitHub連携プラグインやMCPと組み合わせて、AIに調査、比較、候補抽出を任せる

ただし、GitHubにログインするときは、必要以上に広い権限を与えないようにします。仕事用や顧客用の非公開リポジトリをAIに読ませる前には、対象範囲と実行するコマンドを必ず確認してください。

自分がよくやるのは、「直近1か月でスター数が伸びているリポジトリを探して」とAIに頼む使い方です。

<div className="copy-block">
<pre>{`GitHubで、直近1か月にスター数が伸びているAI開発系リポジトリを探してください。
対象は、今やりたい作業に使えそうなスキル、エージェント、ブラウザ自動化、資料処理、開発補助ツールです。

スター数が多いものを優先しつつ、企業公式リポジトリやメンテナンスが続いているものを上にしてください。
各候補について、何に使えるか、初心者が真似できる部分、注意点を短く整理してください。`}</pre>
</div>

コマンドで探すなら、たとえば次のような形です。

<div className="copy-block">
<pre>{`$since = (Get-Date).AddMonths(-1).ToString("yyyy-MM-dd")
gh search repos "AI agent created:>$since" --sort stars --order desc --limit 20`}</pre>
</div>

目的は、流行を追いかけることではありません。自分で考えるステップ、調べるステップ、実装するステップをできるだけ飛ばすことです。

## 自分をAIだと思って考える

もう一つ大事にしているのは、自分のことをAIだと思って考えることです。

普通は、人間がAIに「人間のように考えて」と指示します。ここではあえて順番を逆にして、自分の思考がどのような仕組みで対象を捉えているのかを意識します。

参考になるのは、フッサールの現象学です。

AIに本当に意識があるかどうかは、ここでは一度置いておきます。指示を出すときには、仮に「AIの意識表象の中に対象が立ち上がる」と考えた方が扱いやすいです。

たとえば、自分が見ている画面、気になっている違和感、完成形として想像している雰囲気を、ただ「いい感じにして」と言うだけでは、AIは別の角度から対象を捉えてしまいます。

そうではなく、自分の目線をそのまま移植するつもりで伝えます。

- どこを見ているのか
- 何を同じ対象だと捉えているのか
- 何を違和感だと感じているのか
- どの観点を優先しているのか
- どこから先はまだ判断しなくてよいのか

この目線の共有がうまくいくと、AIは対面で議論する相手というより、自分の脳の拡張として動きやすくなります。

<div className="copy-block">
<pre>{`今から伝える内容は、一般論として広げず、私がいま見ている対象を同じ目線で捉えるための前提として読んでください。

私が見ている対象:
-

違和感を持っている点:
-

優先したい観点:
-

まだ判断しなくてよいこと:
-

この目線を保ったまま、次に何を直すべきかを整理してください。`}</pre>
</div>

## いきなりコードを書かせない

エンジニアではない人が、いきなり「バイブコーディングでこういうサイトを作って」と頼むと、AIはその場でコードを書き始めることがあります。

AIはいろいろな開発手法を知っています。ただし、その場で常に最適な進め方を選んでくれるとは限りません。

いきなり作り始めると、次の問題が起きやすくなります。

1. 思い描いていた完成形と違うものができる
2. 少しずつ進める中で、進捗管理がしにくくなる
3. メンテナンスしにくいコードになる

これを避けるには、最初からプロのエンジニアが使うワークフローに乗せます。

- TDD: 先に期待する動きやテストを書いてから作る
- ドキュメント駆動開発: 先に仕様、使い方、判断基準を書いてから作る
- PRD/要件メモ: 作るもの、やること、やらないこと、完成条件を決める
- レビュー駆動: 作ったあとに、差分、テスト、リスクを見てもらう

難しく考えなくても、「プロのワークフローに則って進めてください」と最初に言うだけでも効果があります。

<div className="copy-block">
<pre>{`いきなりコードを書き始めないでください。
まず、プロの開発ワークフローに則って進め方を決めてください。

次の順番でお願いします。
1. 目的と完成条件を整理する
2. 今回やること、やらないことを分ける
3. 既存の類似ツールや実装例を探す
4. TDD、ドキュメント駆動開発、通常の小さな実装のどれで進めるべきか提案する
5. 最初の小さな作業単位だけ決める

この段階では、まだファイル編集をしないでください。`}</pre>
</div>

## AIが考える余地を減らす

AIには考えさせるべきところと、考えさせない方がよいところがあります。

毎回すべてのデータをAIに読ませて、一つずつ判断させると、トークンを大量に使います。さらに、AIの推論が一度ずれると、そのまま誤った結果が出力されます。

機械的にできる部分は、なるべくスクリプトにします。

- ファイル一覧を作る
- PDFやスライドから文字を抽出する
- 画像だけをページごとに切り出す
- JSONやCSVを検査する
- リンク切れを確認する
- テストやビルドを実行する
- 変更されたファイルだけを一覧化する

Pythonのスクリプトで処理できる部分は、再利用可能な形で保管しておきます。AIには、そのスクリプトを実行させ、出力結果を次の処理に渡させます。

グローバルルールとしては、次のように書いておくと扱いやすいです。

<div className="copy-block">
<pre>{`機械的に判定できる作業は、なるべくAIの推論で処理しないでください。
既存スクリプト、テスト、lint、grep、JSONパース、ハッシュ確認、ブラウザ検証など、決定的な方法を優先してください。
同じ処理を繰り返す場合は、再利用できるスクリプトやチェックリストとして保存してください。`}</pre>
</div>

## 公式ドキュメントを吸収する

AIを開発している企業が公式に出しているドキュメントには、かなり大事なことが入っています。

特に [AnthropicのPrompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) や、[context engineeringの記事](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) は、AIに何を渡すか、どのように作業状態を保つかを考える上で参考になります。

ただし、自分で全部読む必要はありません。

<div className="copy-block">
<pre>{`次の公式ドキュメントを読んで、私のCodex/Claude環境で使えるルールに変換してください。

やってほしいこと:
- 重要な原則を10個以内に圧縮する
- AGENTS.md または CLAUDE.md に入れるべきルールに変換する
- 毎回使えるチェックリストにする
- すでに自分の環境にあるルールと重複するものは統合する
- 実行コマンドや検証方法がある場合は、具体的に残す`}</pre>
</div>

Google Cloudが発表している [Open Knowledge Format](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing) も、この方向性に合っています。OKFは、MarkdownにYAMLフロントマターを付け、人間とAIエージェントが同じ知識を読めるようにする形式です。仕様やサンプルは [GoogleCloudPlatform/knowledge-catalog の okf ディレクトリ](https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf) にあります。

これも、自分で細かい書式を覚える必要はありません。

<div className="copy-block">
<pre>{`このメモを、OKFの考え方に近い形でMarkdown化してください。
YAMLフロントマターには、title、description、tags、resource、timestampを入れてください。
本文は、人間が読んでも分かる説明と、AIが次回参照しやすいチェックリストに分けてください。`}</pre>
</div>
