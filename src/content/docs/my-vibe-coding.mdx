<p className="lead">これは、バイブコーディングをするときに自分が大事にしている個人的な感覚です。最初から一人で全部やろうとせず、とにかく楽をする方法を考えます。</p>

<p className="skip-note">このページは応用・参考です。はじめて読む人は、先に「はじめの一歩」「場面別の頼み方」「フォルダ構成」「戻し方」だけ読めば大丈夫です。</p>

## まず楽をする

バイブコーディングで大事なのは、いきなり自分だけで作り切ろうとしないことです。

- 誰かが作っているものを真似る
- 既存の公開スキルやエージェントを使う
- オープンソースのツールを使う
- 企業が公式に出しているドキュメントをAIに読ませる
- 繰り返す作業はスクリプトにして保管する

探す場所として一番使いやすいのは、やはりGitHubです。Google検索で自力で探し回るより、まず [GitHub CLI](https://cli.github.com/) を入れて、CodexやClaudeから `gh` コマンドを使える状態にしておく方が楽です。

GitHub CLIを使えるようにしておくと、たとえば次のことができます。

1. `gh repo clone` でGitHub上のリポジトリを簡単に手元へ持ってくる
2. `gh search repos` で、条件に合うリポジトリをAIに探させる
3. GitHub連携プラグインやMCPと組み合わせて、AIに調査、比較、候補抽出を任せる

ただし、GitHubにログインするときは、必要以上に広い権限を与えないようにします。仕事用や顧客用の非公開リポジトリをAIに読ませる前には、対象範囲と実行するコマンドを必ず確認してください。

自分がよくやるのは、「直近1か月でスター数が伸びているリポジトリを探して」とAIに頼む使い方です。

<div className="copy-block">
<pre>{`GitHubで、直近1か月にスター数が伸びているAI開発系リポジトリを探してください。
対象は、今やりたい作業に使えそうなスキル、エージェント、ブラウザ自動化、資料処理、開発補助ツールです。

スター数が多いものを優先しつつ、企業公式リポジトリやメンテナンスが続いているものを上にしてください。
各候補について、何に使えるか、初心者が真似できる部分、注意点を短く整理してください。`}</pre>
</div>

コマンドで探すなら、たとえば次のような形です。

<div className="copy-block">
<pre>{`$since = (Get-Date).AddMonths(-1).ToString("yyyy-MM-dd")
gh search repos "AI agent created:>$since" --sort stars --order desc --limit 20`}</pre>
</div>

目的は、流行を追いかけることではありません。自分で考えるステップ、調べるステップ、実装するステップをできるだけ飛ばすことです。

## 自分をAIだと思って考える

もう一つ大事にしているのは、自分のことをAIだと思って考えることです。

普通は、人間がAIに「人間のように考えて」と指示します。ここではあえて順番を逆にして、自分の思考がどのような仕組みで対象を捉えているのかを意識します。

参考になるのは、フッサールの現象学です。

AIに本当に意識があるかどうかは、ここでは一度置いておきます。指示を出すときには、仮に「AIの意識表象の中に対象が立ち上がる」と考えた方が扱いやすいです。

たとえば、自分が見ている画面、気になっている違和感、完成形として想像している雰囲気を、ただ「いい感じにして」と言うだけでは、AIは別の角度から対象を捉えてしまいます。

そうではなく、自分の目線をそのまま移植するつもりで伝えます。

- どこを見ているのか
- 何を同じ対象だと捉えているのか
- 何を違和感だと感じているのか
- どの観点を優先しているのか
- どこから先はまだ判断しなくてよいのか

この目線の共有がうまくいくと、AIは対面で議論する相手というより、自分の脳の拡張として動きやすくなります。

<div className="copy-block">
<pre>{`今から伝える内容は、一般論として広げず、私がいま見ている対象を同じ目線で捉えるための前提として読んでください。

私が見ている対象:
-

違和感を持っている点:
-

優先したい観点:
-

まだ判断しなくてよいこと:
-

この目線を保ったまま、次に何を直すべきかを整理してください。`}</pre>
</div>

## いきなりコードを書かせない

エンジニアではない人が、いきなり「バイブコーディングでこういうサイトを作って」と頼むと、AIはその場でコードを書き始めることがあります。

AIはいろいろな開発手法を知っています。ただし、その場で常に最適な進め方を選んでくれるとは限りません。

いきなり作り始めると、次の問題が起きやすくなります。

1. 思い描いていた完成形と違うものができる
2. 少しずつ進める中で、進捗管理がしにくくなる
3. メンテナンスしにくいコードになる

これを避けるには、最初からプロのエンジニアが使うワークフローに乗せます。

- TDD: 先に期待する動きやテストを書いてから作る
- ドキュメント駆動開発: 先に仕様、使い方、判断基準を書いてから作る
- PRD/要件メモ: 作るもの、やること、やらないこと、完成条件を決める
- レビュー駆動: 作ったあとに、差分、テスト、リスクを見てもらう

難しく考えなくても、「プロのワークフローに則って進めてください」と最初に言うだけでも効果があります。

<div className="copy-block">
<pre>{`いきなりコードを書き始めないでください。
まず、プロの開発ワークフローに則って進め方を決めてください。

次の順番でお願いします。
1. 目的と完成条件を整理する
2. 今回やること、やらないことを分ける
3. 既存の類似ツールや実装例を探す
4. TDD、ドキュメント駆動開発、通常の小さな実装のどれで進めるべきか提案する
5. 最初の小さな作業単位だけ決める

この段階では、まだファイル編集をしないでください。`}</pre>
</div>

## AIに考える余地を減らす

AIには考えさせるべきところと、考えさせない方がよいところがあります。

毎回すべてのデータをAIに読ませて、一つずつ判断させると、トークンを大量に使います。さらに、AIの推論が一度ずれると、そのまま誤った結果が出力されます。

機械的にできる部分は、なるべくスクリプトにします。

- ファイル一覧を作る
- PDFやスライドから文字を抽出する
- 画像だけをページごとに切り出す
- JSONやCSVを検査する
- リンク切れを確認する
- テストやビルドを実行する
- 変更されたファイルだけを一覧化する

Pythonのスクリプトで処理できる部分は、再利用可能な形で保管しておきます。AIには、そのスクリプトを実行させ、出力結果を次の処理に渡させます。

グローバルルールとしては、次のように書いておくと扱いやすいです。

<div className="copy-block">
<pre>{`機械的に判定できる作業は、なるべくAIの推論で処理しないでください。
既存スクリプト、テスト、lint、grep、JSONパース、ハッシュ確認、ブラウザ検証など、決定的な方法を優先してください。
同じ処理を繰り返す場合は、再利用できるスクリプトやチェックリストとして保存してください。`}</pre>
</div>

## 公式ドキュメントを吸収する

AIを開発している企業が公式に出しているドキュメントには、かなり大事なことが入っています。

特に [AnthropicのPrompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) や、[context engineeringの記事](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents) は、AIに何を渡すか、どのように作業状態を保つかを考える上で参考になります。

ただし、自分で全部読む必要はありません。

<div className="copy-block">
<pre>{`次の公式ドキュメントを読んで、私のCodex/Claude環境で使えるルールに変換してください。

やってほしいこと:
- 重要な原則を10個以内に圧縮する
- AGENTS.md または CLAUDE.md に入れるべきルールに変換する
- 毎回使えるチェックリストにする
- すでに自分の環境にあるルールと重複するものは統合する
- 実行コマンドや検証方法がある場合は、具体的に残す`}</pre>
</div>

Google Cloudが発表している [Open Knowledge Format](https://cloud.google.com/blog/products/data-analytics/how-the-open-knowledge-format-can-improve-data-sharing) も、この方向性に合っています。OKFは、MarkdownにYAMLフロントマターを付け、人間とAIエージェントが同じ知識を読めるようにする形式です。仕様やサンプルは [GoogleCloudPlatform/knowledge-catalog の okf ディレクトリ](https://github.com/GoogleCloudPlatform/knowledge-catalog/tree/main/okf) にあります。

これも、自分で細かい書式を覚える必要はありません。

<div className="copy-block">
<pre>{`このメモを、OKFの考え方に近い形でMarkdown化してください。
YAMLフロントマターには、title、description、tags、resource、timestampを入れてください。
本文は、人間が読んでも分かる説明と、AIが次回参照しやすいチェックリストに分けてください。`}</pre>
</div>

## よく使うツール

ここからは、普段よく使うツールや、紹介しておきたいリポジトリです。

基本的には、GitHubのスター数が多いもの、企業の公式リポジトリ、メンテナンスが続いているものを優先します。信頼度が高く、AIにも説明させやすいからです。

ここでは、特に初心者が大きくステップを飛ばせるものを優先して紹介します。

### MarkItDown

[microsoft/markitdown](https://github.com/microsoft/markitdown)

MarkItDownは、Microsoft公式のPython製ドキュメント変換ツールです。PDF、PowerPoint、Word、ExcelなどをMarkdownに変換して、LLMやテキスト分析に使いやすくできます。

よく使う理由は、PDFやスライドをAIに直接読ませるより、先にテキストを抽出した方がトークンを節約できるからです。さまざまなファイル形式に対応している点も便利です。

自分の環境では、PDFやPowerPointを読むときには、まずMarkItDownを使うように、`AGENTS.md` や `CLAUDE.md` に書いておくのがよいです。

ただし、MarkItDownだけで全部分かるわけではありません。画像として埋め込まれた図や、スキャンPDFの文字は、そのままでは十分に読めないことがあります。そこで、資料読み取りの流れは次のようにスキル化しておくと安定します。

1. MarkItDownでテキストを抽出する
2. 並行して、ページ画像や図のスクリーンショットを保存する
3. Markdown内のどの位置に、どの図が対応するかを紐付ける
4. 図の内容、画像内の文字、読み取れない部分を別メモとして保存する
5. 最後に、テキスト抽出結果と画像メモを合わせて要約する

### Playwright

[Playwright](https://playwright.dev/)

Playwrightは、Webブラウザを自動操作するためのツールです。Webアプリのテスト、画面確認、クリック操作、フォーム入力、スクリーンショット取得、スマホ幅での表示確認などに使えます。

バイブコーディングでは、AIに「たぶん動いています」と言わせるだけでは弱いです。Playwrightで実際にブラウザを開き、画面を表示し、ボタンを押し、結果が変わるかを確認します。

特に便利なのは、次のような場面です。

- 作ったサイトが実際に表示されるか確認する
- スマホ幅で文字やボタンが崩れていないか見る
- フォーム入力やページ遷移が動くか試す
- スクリーンショットを保存して、見た目の差分を確認する
- 公開前に、主要なリンクや導線を実ブラウザでチェックする

Playwrightの良いところは、Chromium、Firefox、WebKitを扱えることと、ユーザーが見るラベルやボタン名を使って要素を探せることです。AIにブラウザ確認を任せるときも、「画面を見たつもり」ではなく、実際のブラウザ証拠を残しやすくなります。

<div className="copy-block">
<pre>{`Playwrightで実際のブラウザ確認をしてください。
確認すること:
- デスクトップ幅でページが表示される
- スマホ幅で文字やボタンがはみ出さない
- 主要リンクをクリックできる
- スクリーンショットを保存する
- 失敗した場合は、推測ではなく実際のエラーと画面状態を報告する`}</pre>
</div>

### Product Starter

[jujunjun110/product-starter](https://github.com/jujunjun110/product-starter)

Product Starterは、Claude Codeへの指示だけでプロダクトを作っていくためのスターターキットです。Next.js、Supabase、Prisma、shadcn/ui、Tailwind、Vitest、dependency-cruiser、Biomeなどが前提に入っていて、最初から開発の型に乗りやすくなっています。

これを使う理由は、初心者が迷いやすい部分をかなり飛ばせるからです。

- フォルダ構成を一から考えなくてよい
- DDD 4層のような設計ルールを最初から持てる
- よくある作業をスキルやコマンドとして呼び出せる
- テストや依存方向のチェックを、品質ゲートとして使える
- `pnpm verify` のような確認コマンドをAIに実行させやすい

特に、自分で「きれいな設計」や「TDDっぽい進め方」を全部理解してから始めようとすると、そこで止まってしまいます。最初から型があるスターターに乗って、AIにそのルールを守らせる方が楽です。

<div className="copy-block">
<pre>{`jujunjun110/product-starter を参考にして進めてください。
いきなり独自構成を作らず、README、docs、.claude/commands、.claude/skills を読んでください。

やってほしいこと:
- このプロジェクトで使える開発の型を整理する
- 既存の構成に合わせて、真似する部分と真似しない部分を分ける
- テスト、依存方向チェック、verifyコマンドを品質ゲートとして使う
- 最初の実装単位を小さく切る
- コードを書く前に、完成条件と検証方法を決める`}</pre>
</div>

### Information Accessibility Skill and Agent

[Nagi-Inaba/information-accessibility-skill](https://github.com/Nagi-Inaba/information-accessibility-skill)

これは、情報アクセシビリティのレビューや設計に使うためのCodex / Claude向けスキルとエージェントです。Webサイト、アプリ、文書、スライド、動画、イベント案内、会議運営、コミュニティ導線などを、「必要な人が情報を見つけられるか、受け取れるか、理解できるか、参加できるか、後から確認できるか」という観点で確認できます。

これを使う理由は、アクセシビリティのチェックリストを一から作らなくてよくなるからです。

- WebやアプリのUIレビューに使える
- PDF、スライド、告知画像の共有前チェックに使える
- イベントや会議の案内、字幕、文字起こし、資料共有の確認に使える
- CodexとClaudeの両方に入れやすい形で置いておける
- レビュー結果を、優先度、影響を受ける人、修正案、確認方法に分けて出せる

バイブコーディングでは、機能を作ることに意識が寄りがちです。しかし、実際に使う人が情報にたどり着けない、読めない、操作できない、後から確認できない状態だと、プロダクトとしてはかなり弱くなります。

最初からこのスキルを入れておき、AIに「情報アクセシビリティの観点でレビューして」と頼めるようにしておくと、公開前チェックの抜けをかなり減らせます。

<div className="copy-block">
<pre>{`information-accessibility-skill を使って、このページを情報アクセシビリティの観点でレビューしてください。

確認してほしいこと:
- 必要な人が情報を見つけられるか
- テキスト、画像、リンク、フォーム、動画、資料の受け取り方に問題がないか
- 専門用語、日付、場所、参加条件、次の行動が分かりやすいか
- 支援依頼や問い合わせの導線があるか
- 後から資料、要約、記録、次の行動を確認できるか

結果は、Priority / Issue / Who is affected / Fix / Verification の表で出してください。`}</pre>
</div>
