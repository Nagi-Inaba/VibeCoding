<p className="lead">ここでは、普段よく使うツールや、紹介しておきたいリポジトリをまとめます。最初から全部を覚える必要はありません。必要になった場面で、AIにこのページを読ませて使い方を整えてもらえば大丈夫です。</p>

<p className="skip-note">迷ったら、まずは「音声入力ならTypeless」「資料を読むならMarkItDown」「画面を確かめるならPlaywright」「開発の型に乗るならProduct Starter」「公開前の抜けを減らすならアクセシビリティ検査」と覚えておけば十分です。</p>

基本的には、企業の公式リポジトリ、メンテナンスが続いているもの、GitHubで利用者が多いものを候補にします。スター数は品質保証ではなく、見つけるときの目安です。採用前にはREADME、更新日、ライセンス、Issue、セキュリティ上の注意点を確認します。

ここでは、特に初心者が大きくステップを飛ばせるものを優先して紹介します。

## Typeless

[Typeless](https://www.typeless.com/?via=24cb67)

Typelessは、音声入力した内容を自動で整えてくれるツールです。バイブコーディングでは、作りたいものや困っていることを最初からきれいな文章にしようとすると手が止まりやすいので、まず話して材料を出す使い方が楽です。

おすすめの理由は、適当にダラダラと話しても、自動で文章を整形してくれるところです。「えー」「あー」のようなフィラーを減らし、順番がばらばらな話も読みやすい文章や箇条書きに整えてくれるので、あとからAIに渡す下書きとして使いやすくなります。

音声入力は必須ではありません。スマホやPCに最初から入っている音声入力でも試せます。ただ、話した内容をそのまま文字にするだけでなく、AIに渡しやすい形へ整えるところまで任せたいならTypelessが便利です。

向いている場面は、たとえば次のようなときです。

- 作りたいアプリやページのアイデアを出す
- 困っていることを説明する
- 画面を見ながら直したい点を話す
- エラーが出るまでの流れを残す
- 作業後の振り返りを書く

Typelessは外部サービスです。話した内容や文章を送る前に、個人情報、勤務先の内部情報、公開してはいけない資料名が入っていないか確認してください。料金や無料枠は変わることがあるので、使い始める前に現在のプラン内容も見ておくと安心です。

このリンクには紹介用のパラメータが含まれています。気になる場合は、検索して公式サイトから開いても大丈夫です。

使う前に確認すること:

- 公式サイトの料金や無料枠
- 入力した音声や文章の扱い
- 個人情報、勤務先の内部情報、公開してはいけない資料名が入っていないか

Typelessで整ったメモは、そのままAIに渡して大丈夫です。大事なのは、話す順番を自分で完璧に整えることではなく、AIに作業前の確認をさせることです。

<div className="copy-block">
<pre>{`次のメモを読んで、今回の作業を始める前に確認してください。
作りたいもの、今回やること、まだやらないこと、不安なことに分けてください。
この段階では、まだファイル編集はしないでください。`}</pre>
</div>

## MarkItDown

[microsoft/markitdown](https://github.com/microsoft/markitdown)

MarkItDownは、Microsoft公式のPython製ドキュメント変換ツールです。PDF、PowerPoint、Word、ExcelなどをMarkdownに変換して、LLMやテキスト分析に使いやすくできます。

よく使う理由は、PDFやスライドをAIに直接読ませるより、先にテキストを抽出した方がトークンを節約できるからです。さまざまなファイル形式に対応している点も便利です。

ただし、変換したからといって元資料の意味が完全に再現されるわけではありません。機密資料や未確認ファイルを処理する前には、そのファイルを自分の環境で扱ってよいか確認してください。表、図、スキャンPDF、画像内文字は抜けることがあるので、重要な資料では画像メモや目視確認も残します。

自分の環境では、PDFやPowerPointを読むときには、まずMarkItDownを使うように、`AGENTS.md` や `CLAUDE.md` に書いておくのがよいです。

使う前に確認すること:

- 公式READMEで対応ファイルやインストール方法を確認する
- 機密資料や未確認ファイルを処理してよい環境か確認する
- 変換結果だけで判断せず、重要な図表やスキャンPDFは目視でも確認する

ただし、MarkItDownだけで全部分かるわけではありません。画像として埋め込まれた図や、スキャンPDFの文字は、そのままでは十分に読めないことがあります。そこで、資料読み取りの流れは次のようにスキル化しておくと安定します。

1. MarkItDownでテキストを抽出する
2. 並行して、ページ画像や図のスクリーンショットを保存する
3. Markdown内のどの位置に、どの図が対応するかを紐付ける
4. 図の内容、画像内の文字、読み取れない部分を別メモとして保存する
5. 最後に、テキスト抽出結果と画像メモを合わせて要約する

## Playwright

[Playwright](https://playwright.dev/)

Playwrightは、Webブラウザを自動操作するためのツールです。Webアプリのテスト、画面確認、クリック操作、フォーム入力、スクリーンショット取得、スマホ幅での表示確認などに使えます。

バイブコーディングでは、AIに「たぶん動いています」と言わせるだけでは弱いです。Playwrightで実際にブラウザを開き、画面を表示し、ボタンを押し、結果が変わるかを確認します。

特に便利なのは、次のような場面です。

- 作ったサイトが実際に表示されるか確認する
- スマホ幅で文字やボタンが崩れていないか見る
- フォーム入力やページ遷移が動くか試す
- スクリーンショットを保存して、見た目の差分を確認する
- 公開前に、主要なリンクや導線を実ブラウザでチェックする

Playwrightの良いところは、Chromium、Firefox、WebKitを扱えることと、ユーザーが見るラベルやボタン名を使って要素を探せることです。AIにブラウザ確認を任せるときも、「画面を見たつもり」ではなく、実際のブラウザ証拠を残しやすくなります。

使う前に確認すること:

- 確認したいページやURL
- 合格にする条件
- スクリーンショットやログをどこに残すか
- ログインや個人情報を含む画面を扱ってよいか

<div className="copy-block">
<pre>{`Playwrightで実際のブラウザ確認をしてください。
確認すること:
- デスクトップ幅でページが表示される
- スマホ幅で文字やボタンがはみ出さない
- 主要リンクをクリックできる
- スクリーンショットを保存する
- 失敗した場合は、推測ではなく実際のエラーと画面状態を報告する`}</pre>
</div>

## Product Starter

[jujunjun110/product-starter](https://github.com/jujunjun110/product-starter)

Product Starterは、Claude Codeへの指示だけでプロダクトを作っていくためのスターターキットです。Next.js、Supabase、Prisma、shadcn/ui、Tailwind、Vitest、dependency-cruiser、Biomeなどが前提に入っていて、最初から開発の型に乗りやすくなっています。

スターターキットは更新されます。使う前にREADME、最終更新日、依存関係、ライセンスを確認し、いま作っているものにそのまま採用してよいかAIに整理させます。分からないまま全部コピーするより、真似する部分と真似しない部分を分ける方が安全です。

使う前に確認すること:

- 公式READMEと最終更新日
- ライセンス
- 前提になる技術や依存関係
- 自分の目的に必要な部分と、真似しない部分
- セキュリティや公開前確認の仕組みがあるか

これを使う理由は、初心者が迷いやすい部分をかなり飛ばせるからです。

- フォルダ構成を一から考えなくてよい
- DDD 4層のような設計ルールを最初から持てる
- よくある作業をスキルやコマンドとして呼び出せる
- テストや依存方向のチェックを、品質ゲートとして使える
- `pnpm verify` のような確認コマンドをAIに実行させやすい

特に、自分で「きれいな設計」や「TDDっぽい進め方」を全部理解してから始めようとすると、そこで止まってしまいます。最初から型があるスターターに乗って、AIにそのルールを守らせる方が楽です。

<div className="copy-block">
<pre>{`jujunjun110/product-starter を参考にして進めてください。
いきなり独自構成を作らず、README、docs、.claude/commands、.claude/skills を読んでください。

やってほしいこと:
- このプロジェクトで使える開発の型を整理する
- 既存の構成に合わせて、真似する部分と真似しない部分を分ける
- テスト、依存方向チェック、verifyコマンドを品質ゲートとして使う
- 最初の実装単位を小さく切る
- コードを書く前に、完成条件と検証方法を決める`}</pre>
</div>

## 情報アクセシビリティの検査サブエージェント

[Nagi-Inaba/information-accessibility-skill](https://github.com/Nagi-Inaba/information-accessibility-skill)

これは、情報アクセシビリティのレビューや設計に使うためのCodex / Claude向けスキルとエージェントです。Webサイト、アプリ、文書、スライド、動画、イベント案内、会議運営、コミュニティ導線などを、「必要な人が情報を見つけられるか、受け取れるか、理解できるか、参加できるか、後から確認できるか」という観点で確認できます。

これを使う理由は、アクセシビリティのチェックリストを一から作らなくてよくなるからです。

使う前に確認すること:

- CodexやClaudeにスキル/サブエージェントを入れているか
- 入れていない場合は、下の通常AI向けプロンプトを使う
- レビュー対象に個人情報や内部資料が入っていないか
- 指摘をどのページや作業記録に残すか

- WebやアプリのUIレビューに使える
- PDF、スライド、告知画像の共有前チェックに使える
- イベントや会議の案内、字幕、文字起こし、資料共有の確認に使える
- CodexとClaudeの両方に入れやすい形で置いておける
- レビュー結果を、優先度、影響を受ける人、修正案、確認方法に分けて出せる

バイブコーディングでは、機能を作ることに意識が寄りがちです。しかし、実際に使う人が情報にたどり着けない、読めない、操作できない、後から確認できない状態だと、プロダクトとしてはかなり弱くなります。

最初からこのスキルを入れておくと、公開前チェックの抜けをかなり減らせます。頼むときは、長い条件を書かなくても大丈夫です。

普通のAIチャットしかない場合は、次のように頼めば十分です。

<div className="copy-block">
<pre>{`このページを、情報アクセシビリティの観点でレビューしてください。
Find / Receive / Understand / Participate / Continue の5つに分けて見てください。

確認してほしいこと:
- 初心者が必要な情報を見つけられるか
- 文字、画面、スマホ、読み上げで受け取りにくい情報がないか
- 誤解しそうな表現や長すぎる説明がないか
- 困ったときに相談しやすいか
- 後から作業を再開しやすいか

問題点、影響を受ける人、直し方、確認方法を表で出してください。`}</pre>
</div>

CodexやClaudeにサブエージェントを入れている場合は、もっと短く頼めます。

<div className="copy-block">
<pre>{`アクセシビリティのサブエージェントで検査して。`}</pre>
</div>
